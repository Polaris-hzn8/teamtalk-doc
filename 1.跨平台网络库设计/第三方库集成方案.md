# 第三方库集成方案

---

C++项目中会集成很多的第三方库，

其他的业务服务如teamtalk-server、会依赖第三方库、以及新增加的teamtalk-imcore编译出来的动态库。那么如何设计寻找动态库的路径是个问题，是否在某个目录下建立统一的动态库位置集成多个开源库？

<mark>主流做法是：workspace级别的sdk目录</mark> 

在仓库根目录下统一建立sdk目录，imcore编译后install将编译结果安装进去，其他子项目从其中查找对应的依赖：

```
teamtalk/
├── sdk/
│   ├── include/
│   │   └── imcore/         ← imcore 头文件
│   └── lib/
│       ├── libimcore.so
│       └── libprotobuf.a
├── third/                  ← 第三方源码或预编译包（不变）
├── teamtalk-imcore/        ← cmake install → ../sdk/
├── teamtalk-server/        ← find_package 从 ../sdk/ 找
└── teamtalk-win/           ← 同上
```

```cmake
# teamtalk-imcore/CMakeLists.txt
install(TARGETS imcore
    ARCHIVE DESTINATION ${CMAKE_SOURCE_DIR}/../sdk/lib
    LIBRARY DESTINATION ${CMAKE_SOURCE_DIR}/../sdk/lib
)
install(DIRECTORY src/ DESTINATION ${CMAKE_SOURCE_DIR}/../sdk/include/imcore
    FILES_MATCHING PATTERN "*.h"
)
# teamtalk-server/CMakeLists.txt
set(IMCORE_DIR ${CMAKE_SOURCE_DIR}/../sdk)
include_directories(${IMCORE_DIR}/include)
link_directories(${IMCORE_DIR}/lib/linux)
target_link_libraries(msg_server imcore protobuf ...)
```

```shell
# 软件打包
release/
├── bin/
│   ├── msg_server
│   ├── login_server
│   ├── db_proxy_server
│   └── route_server
├── lib/
│   └── libimcore.so        ← 如果用动态库
├── conf/
│   ├── msg_server.conf
│   ├── login_server.conf
│   └── log4cxx.properties
├── scripts/
│   ├── start_all.sh
│   └── stop_all.sh
└── sql/
    └── init.sql
```
