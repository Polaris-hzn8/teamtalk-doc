# 第三方库集成方案

---

C++项目中会集成很多的第三方库，

其他的业务服务如teamtalk-server、会依赖第三方库、以及新增加的teamtalk-imcore编译出来的动态库。那么如何设计寻找动态库的路径是个问题，是否在某个目录下建立统一的动态库位置集成多个开源库？

### workspace级别的.sdk目录

<mark>主流做法是：workspace级别的sdk目录</mark> 

在仓库根目录下统一建立sdk目录，imcore编译后install将编译结果安装进去，其他子项目从其中查找对应的依赖：

```
teamtalk/
├── .sdk/{repo_name}/{version}
│   ├── include/			← 第三方库头文件
│   └── lib/				← 第三方动态库.so
├── third/                  ← 第三方源码或预编译包（不变）
├── teamtalk-imcore/        ← cmake install → ../sdk/
├── teamtalk-server/        ← find_package 从 ../sdk/ 找
└── teamtalk-win/           ← 同上
```

```cmake
# teamtalk-imcore/CMakeLists.txt
install(TARGETS imcore
    ARCHIVE DESTINATION ${CMAKE_SOURCE_DIR}/../sdk/lib
    LIBRARY DESTINATION ${CMAKE_SOURCE_DIR}/../sdk/lib
)
install(DIRECTORY src/ DESTINATION ${CMAKE_SOURCE_DIR}/../sdk/include/imcore
    FILES_MATCHING PATTERN "*.h"
)
# teamtalk-server/CMakeLists.txt
set(IMCORE_DIR ${CMAKE_SOURCE_DIR}/../sdk)
include_directories(${IMCORE_DIR}/include)
link_directories(${IMCORE_DIR}/lib/linux)
target_link_libraries(msg_server imcore protobuf ...)
```

```shell
# 软件打包
release/
├── bin/
│   ├── msg_server
│   ├── login_server
│   ├── db_proxy_server
│   └── route_server
├── lib/
│   └── libimcore.so        ← 如果用动态库
├── conf/
│   ├── msg_server.conf
│   ├── login_server.conf
│   └── log4cxx.properties
├── scripts/
│   ├── start_all.sh
│   └── stop_all.sh
└── sql/
    └── init.sql
```



### 参考集成方案

yaml版号设置如下：

```yaml
name: blitz/gateway_serv
version: 1.0.0.80
template: cmake
ugbase: ssh://git@gitlab.ugnas.com/scm/ugbase.git master 69cfa0fedb9f6539e3c540d8708e94dce3b221c2
platform:
- build_type
- cpu_arch
output:
  format: both
release: dev
release-name: 网关
deps:
- blitz/ugos-core@release_blitz(UGCORE) 1.0.0.147
- blitz/3rd/seastar@release_25.5.0 25.5.0.33
- blitz/3rd/jemalloc@release_4.5.1 4.5.1.2
- devtools/ugtools@release_v2 1.1.0.8
```

1. 为每个集成的仓库设定版号：动态库、静态库、或二进制，都设置版本号，
2. 构建系统构建时读取仓库相关的yaml文件，确认对应的版本依赖是否已经存在，
   - 如果否则从远端下载到对应的版号以及头文件
   - 如果所有的依赖项都在则进行下一步编译动作
3. 生产当前的编译结果，并将编译结果上传的远端仓库中
4. 针对第三方仓库，将第三方仓库设置版本号将第三方仓库编译并将结果，下载到本地sdk中
5. 在软件打包的过程中，将所有依赖的动态库打包到软件包中。

#### ugbt构建工具分析

```
~/.ugreen.tmp/repo/{name}/{branch}/{version}/{build_number}/{platform}/
  └── {pack_name}/
       ├── include/
       ├── lib/      ← .so 动态库
       └── cmake/    ← find_package 用的 cmake 配置
```

编译c++服务完整链路，

1. `ugbt build` → `UgProject.gen_makefiles()` 生成两个文件：
   - `ugreen_build/Makefile-{platform}`（主框架）
   - `ugreen_build/cmake-{platform}.mk`（cmake 子流程）
2. `make prepare` → `__process_dep()` 递归下载所有依赖（含传递依赖），创建符号链接
3. `cmake` 执行时先调用 `ugbt gen-ugbt-cmake` 生成 `ugbt.cmake`，把每个依赖的 cmake 目录注入 `CMAKE_MODULE_PATH` 和 `CMAKE_PREFIX_PATH`
4. C++ 工程的 `CMakeLists.txt` 只需 `include(ugbt.cmake)` + `ugbt_init()` 即可让 `find_package()` 找到所有库
5. `source ug_env.sh` 在编译期注入所有依赖的路径环境变量（`{NAME}_ROOT`、`LD_LIBRARY_PATH` 等）

#### find_package相关

1. ugbt gen-ugbt-cmake生产的文件是什么？有什么作用？
2. ug_env.sh的生成规则是什么？有什么作用？

ugbase/CMake流程中，find_package之所以能看到拉到本地的依赖，核心是 `ugbt.cmake` + `CMAKE_MODULE_PATH` / `CMAKE_PREFIX_PATH`。

<mark>ugbt为find_package铺路</mark>：

`UgProject.generate_cmake()` 会为每个 ugbase 依赖（含隐式依赖）生成一段 CMake：把
`${CMAKE_BINARY_DIR}/deps/<名字转义>/cmake` 插到 `CMAKE_MODULE_PATH` 和 `CMAKE_PREFIX_PATH` 前面：

- `CMAKE_BINARY_DIR`/deps/`…`/cmake：对应 prepare 时在 `ugreen_build/<platform>/deps/` 下链过去的每个包里带的 `cmake/`（常见于 `FooConfig.cmake` 或配套的 `FindFoo.cmake` 风格）。
- `find_package(Mod ...)`：
  - Module 模式：在 `CMAKE_MODULE_PATH` 里找 `FindMod.cmake`。
  - CONFIG 模式：在 prefix 下找 `<lowercase>-config.cmake` / `ModConfig.cmake`，也就是 `CMAKE_PREFIX_PATH` 要能指到装了这些文件的目录——这里写的是 `deps/xxx/cmake`，和你们包内布局一致时就能命中。

<mark>工程中的使用</mark>：

这套仓库自带的 cmake/ugreen_common.cmake 里是典型写法：

顺序很重要：必须先 `include(ugbt.cmake)` 再 `ugbt_init()`，然后后面的 `find_package(...)` 才会在扩充过的搜索路径里解析。

同一个文件里也混合了两种方式：

- Module / 手写 Find：例如 `find_package(Boost REQUIRED)`（仓库里还带了自己的 `cmake/FindBoost.cmake`）。
- CONFIG：例如 `find_package(gRPC REQUIRED CONFIG)`，依赖 CMake 的配置包和标准 prefix 查找。

另外还有 环境变量（如 `Boost` 用 `$ENV{BOOST_ROOT}`），和 `PATH`/`LD_LIBRARY_PATH`（`ug_env.sh`）更多是编译、链接和运行时用，不完全替代 `find_package` 的包发现逻辑。

<mark>如果需要链接自己的动态库</mark>：

要保证 `find_package(YourLib ...)` 能工作，你们的依赖包里通常需要 `deps/<xxx>/cmake` 下放好：

- `YourLibConfig.cmake`（推荐 CONFIG），或
- `FindYourLib.cmake`（Module），且包名与 `find_package` 一致。

这样既符合 ugbase 生成的路径约定，也最符合 CMake 的常规用法。
