# 跨平台通信库抽离imcore

---

问题背景：目前teamtalk-server中的imcore与teamtalk-win中的ImCore中存在多处重复代码逻辑，

1. 希望将这些逻辑封装成单独的一个同时支持Windows与Linux平台的跨平台网络基础库 imcore，
2. 并且后续所有的基础封装代码都将增加到这个库中。库中的代码可以同时提供给server后端与Windows客户端同时使用。
3. 多个平台下的pb通信协议定义希望集成到该库中，而不是每个平台独立维护一份，降低代码维护与同步成本。

### imcore定位

客户端与服务器端，共用网络层 + 协议层 + 纯工具类：

1. imcore存在的意义就是为了让客户端和服务端之间，可以互相通信。如果两边各自维护协议修改后迟早出现问题bug。

2. 客户端与服务端共用的纯工具类放在imcore中，也没有任何问题等（这些没有平台假设、没有业务逻辑、就是数学和字符串操作，两端用同份代码）

   ```cpp
   security（AES/Base64/MD5）
   string 工具（URLEncode、memfind）
   singleton 模板
   jsoncpp
   ```

但是需要明确不是所有基础库都放到imcore中来：

这会让imcore变成大杂烩，尤其是客户端与服务端对基础设施的需求差异很大：

| 需求     | 服务端                | 客户端                    |
| :------- | :-------------------- | :------------------------ |
| 线程模型 | 多线程/线程池         | 单 UI 线程 + 少量后台线程 |
| 配置文件 | `.conf` 服务配置      | 注册表 / 用户偏好         |
| HTTP     | 作为服务对外提供 HTTP | 作为客户端发 HTTP 请求    |
| 数据库   | MySQL/Redis 访问      | 本地 SQLite 缓存          |
| 日志     | 高并发异步写文件      | 调试输出 + 少量文件       |

业界的处理方式：imcore 只放两端真正共用的东西，各端的基础设施在各自的仓库中维护。

```
┌─────────────────────────────┐
│        imcore（网络通信 + 协议）│  ← 协议、网络、纯工具
└─────────────────────────────┘
         ↑               ↑
┌──────────────┐  ┌──────────────┐
│  server-base │  │  client-base │  ← 各自的基础设施
│  线程池/DB代理│  │  UI工具/缓存  │
└──────────────┘  └──────────────┘
```

### 1.日志库整合

#### 现有方案

Windows下使用yaolog进行日志输出，linux下封装log4cxx来进行日志输出，这是导致跨平台公共库封装的差异点之一，有限需要解决这个问题，否则无法正常进行日志输出。当前日志库现状如下：

| 位置              | 使用的库                           | 调用方式                          |
| :---------------- | :--------------------------------- | :-------------------------------- |
| `teamtalk-server` | log4cxx（via `slog_api.h`）+ log4z | `CSLog::Info()`、`CSLog::Error()` |
| `teamtalk-win`    | yaolog（2013年，已停更）           | yaolog 自身宏                     |

<mark>log4cxx</mark>：

1. Apache log4cxx 现在有 CI 流水线跑 Windows（MSVC + MinGW 均支持
2. 但log4cxx 强依赖 Apache APR，在 Windows 上的根本问题
3. 对于桌面IM客户端，引入APR这个量级的依赖只是为了打印日志代价过重。加上 MFC 项目集成 APR 的 CMake 体系有额外摩擦，工程复杂度明显上升。

<mark>yaolog</mark>：

1. yaolog 2013 年停更，功能够用但无人维护，放弃完全合理。

#### 替换方案spdlog

放弃 yaolog 完全正确。统一日志首选 spdlog 而不是 log4cxx。log4cxx 的 APR 依赖对桌面客户端是不必要的负担，spdlog 能以接近零成本实现真正的跨平台统一。日志库选型替换：spdlog，推荐方案：日志接口层+spdlog作为统一日志输出模式。

| 对比         | log4cxx               | spdlog                     |
| :----------- | :-------------------- | :------------------------- |
| 外部依赖     | APR + APR-util（重）  | 无（header-only 可用）     |
| Windows 支持 | 可用但有 DLL 问题     | 原生，零问题               |
| CMake 集成   | 复杂                  | `FetchContent` 一行        |
| 性能         | 一般                  | 极快（异步模式 ~1M msg/s） |
| 维护状态     | Apache 维护，但发版慢 | 活跃维护，2025 年仍有更新  |
| 分发         | 需附带多个 .dll       | 静态编译无额外文件         |

核心并发能力对比：

| 特性               | log4cxx（当前）            | spdlog                                      |
| :----------------- | :------------------------- | :------------------------------------------ |
| 线程安全           | ✅ 互斥锁保护               | ✅ 可选互斥锁或无锁                          |
| 异步写入           | ✅ AsyncAppender            | ✅ async logger（lock-free 队列 + 后台线程） |
| 吞吐量（异步模式） | ~50万条/秒                 | ~200-500万条/秒                             |
| 调用线程阻塞       | 异步模式下不阻塞           | 异步模式下不阻塞                            |
| 滚动日志文件       | ✅ RollingFileAppender      | ✅ rotating_file_sink                        |
| 按时间切割         | ✅ DailyRollingFileAppender | ✅ daily_file_sink                           |

TeamTalk 服务端架构目前是多进程 + 单线程事件循环（netlib），只有db_proxy_server有线程池。实际日志压力远低于spdlog的性能上限，spdlog 异步模式绰绰有余。log4cxx有个spdlog没有的特性：spdlog 没有热重载机制。

```cpp
// slog_api.cpp
// 运行时热重载日志级别，每10秒检查配置文件变更
PropertyConfigurator::configureAndWatch("log4cxx.properties", 10000);
```

#### 差异点补齐

##### 日志热重载

spdlog没有内置热重载，但是可以自己实现且成本较低：

<mark>方案A</mark>：利用信号触发配置重载（Linux服务端标准做法）比 log4cxx 的轮询文件方式更可控，避免后台线程持续 I/O。

```cpp
// 收到 SIGUSR1 → 重新读取配置文件 → 调整 spdlog 日志级别
signal(SIGUSR1, [](int) {
    reloadLogConfig();  // 读 log.conf，调用 spdlog::set_level()
});
```

```shell
# 运维操作：
kill -USR1 <pid>   // 触发热重载，无需重启，无连接中断
```

<mark>方案B</mark>：轮询文件（与log4cxx行为相同）

```cpp
// 后台线程每 10 秒检查 log.conf 的 mtime
// 变化则重新解析并调用 spdlog::set_level()
```

##### 客户端日志模式调整兼容

yaolog整合问题点：

1. yaolog的 `LOG__` 日志打印是频道日志，而不是级别式的日志。`NET/APP/ERR/DEBG/SOCK` 
2. yaolog直接支持二进制流式数据打印，
   - 这对网络协议调试非常有价值（可以直接看到Protobuff包的原始字节）
   - 而spdlog没有内置的hex dump需要自己实现转换函数，然后再传递给spdlog打印出来
3. yaolog原生支持Unicode/ANSI双模式，spdlog针对wchar_t的支持有限，在Windows下的TCHAR场景需要显式转换。

| 场景                                   | 替换 spdlog 成本                          |
| :------------------------------------- | :---------------------------------------- |
| TTClient 的 `LOG__(channel, ...)` 调用 | 高：100+ 处调用，设计模型不同，需重新封装 |
| 二进制日志 `LOGBIN__`                  | 中：需自己实现 hex dump 工具函数          |
| TCHAR 兼容                             | 中：需要处处加 wchar→char 转换            |
